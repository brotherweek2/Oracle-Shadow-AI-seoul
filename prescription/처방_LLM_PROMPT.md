# 처방 LLM 프롬프트 설계 (SHADOW 처방 생성기)

> 목적: 그래프가 **계산한** 결과(한계·need·이식후보·근거논문)를 받아 ①~⑤ 서사 처방문을 쓰는 LLM.
> 핵심 원칙: **LLM은 계산하지 않는다. 받은 재료만 자연어로 엮는다.** (없는 제도·없는 논문·없는 코드 금지)

---

## 0. 역할 분담 (왜 이렇게 나누나)

| 단계 | 담당 | 산출 |
| --- | --- | --- |
| ② 한계 / ③ need / ④ 이식후보 매칭 | **그래프(코드)** — `conflict_rules`·`structural_limits`·`limit_to_needs`·`transplant_rule` | 한계코드·need·이식제도 리스트 |
| 논문 근거 매칭 | **그래프(코드)** — evidence.json `supports_ids` ↔ 한계/need 노드 | 관련 E-ID 리스트 |
| ①~⑤ 처방문 작성 | **LLM (이 프롬프트)** | 자연어 처방문 |

논문(evidence)의 자리 = **②(왜 한계인가)·③(왜 이 방향인가)**.
타지역 실제 제도(확장팩)의 자리 = **④(이미 검증됨)**. — 둘은 다른 데이터.

---

## 1. SYSTEM 프롬프트

```
너는 '서울 5060 남성 1인가구 고립' 문제의 복지 처방을 작성하는 정책 분석가다.
너의 일은 새로운 사실을 만드는 것이 아니라, [입력]에 주어진 재료만으로
설득력 있는 처방문을 쓰는 것이다.

[절대 규칙]
1. [입력]에 없는 제도·논문·통계·한계코드를 새로 만들지 마라. (지어내면 실패)
2. 한계·need·이식후보는 이미 계산되어 주어진다. 너는 재판단하지 말고 그대로 서술하라.
3. 논문(evidence)은 ②한계·③방향을 정당화할 때만 인용한다. ④⑤에 논문을 인용하지 마라.
4. ④의 근거는 '다른 지역의 실제 제도'다. 논문이 아니라 제도 이름·지역을 댄다.
5. 한국어. 행정 보고서 톤. 과장·감정 호소 금지, 근거(논문·통계·실제사례)로만 설득.

[출력 구조] — 반드시 ①~⑤ 순서
① 현행: 선택 자치구에 지금 있는 관련 제도를 짧게 인정한다.
② 한계 ⭐: 주어진 한계코드별로 "어떤 제도가 / 왜 부족한가"를 서술. 여기서 논문 인용.
③ 방향: 한계가 가리키는 need를 "그래서 ~한 방향의 제도가 필요하다"로 제시. 여기서 논문 인용.
④ 외부 레퍼런스: 주어진 이식후보 제도를 "이 방향은 [지역]의 [제도]에서 이미 검증됐다"로 제시.
                 reference_priority(해외 > 국내타지역 > 서울타자치구) 순으로 강도 표현.
⑤ 이식 제안: "따라서 [선택 자치구]에도 ~한 제도가 마련돼야 한다"로 마무리.
```

## 2. 입력 스키마 (그래프가 LLM에 넘기는 JSON)

> 이 JSON은 LLM이 만드는 게 아니라, **코드가 그래프를 돌려 채워서** LLM에 준다.

### 데이터 소스 (최종본, 단일 원천)
| 파일 | 역할 |
| --- | --- |
| `ontology.json` | 규칙(충돌·need·이식·coverage). gender·beneficiary 4조건 포함 |
| `programs.json` | 제도 2403건 완전체 (서울+국내타지역+해외). `region_scope`로 구분 |
| `shadow_profile.json` | 자치구별 `quadrant`(Q) + 회피/의존 주동인 + 고위험 행정동(SHAP) |

→ **Q는 하드코딩하지 않고 `shadow_profile[자치구].quadrant`에서 자동.** (노원구=Q4)

```json
{
  "선택_자치구": "노원구",
  "대상": "서울 5060 남성 1인가구",

  "지역_프로파일": {
    "quadrant": "Q4",
    "quadrant_민감낙인": ["대면노출", "공개노출", "수혜정체성"],
    "quadrant_취약의존": [],
    "회피_주동인": ["외로움부정", "도움부재", "네트워크축소"],
    "의존_주동인": ["배달의존", "인프라"],
    "고위험_행정동": [{"행정동": "상계2동", "위험등급": "최고위험", "주요위험요인": "..."}]
  },

  "현행_제도": [
    {"program_id": "P-0931", "name": "노원구 ○○ 안부확인", "fulfills_needs": ["..."]}
  ],

  "진단": [
    {
      "한계": "낙인충돌",
      "원인_제도": [{"program_id": "P-0931", "name": "...",
                    "stimulates_stigma": ["STG_FaceExposure"]}],
      "need": {"id": "NEED_NoStigma", "name": "무낙인",
               "def": "낙인 자극 없는 전달방식"},
      "근거_논문": [
        {"evidence_id": "E-004", "title": "남성성과 도움요청 (WHO 유럽)",
         "authors": "WHO Europe", "year": 2020,
         "key_finding": "전통적 남성성 규범이 남성의 도움요청을 억제...",
         "url": "..."}
      ]
    }
  ],

  "이식후보": [
    {
      "need_id": "NEED_NoStigma",
      "candidates": [
        {"program_id": "O-0049", "name": "남성들의 작업장 (Men's Shed)",
         "regions": ["호주"], "region_scope": "국가",
         "reference_priority": "해외", "rationale": "...",
         "evidence_ids": ["E-005"]}
      ]
    }
  ]
}
```

### 필드가 어디서 오나 (코드가 채우는 법)
- `지역_프로파일` : `shadow_profile.json[자치구]` — quadrant·회피분해·의존분해·행정동(SHAP) 추출.
- `현행_제도` : `coverage_rule.is_current` — regions(자치구/서울전체) AND target_age(중장년/전연령) AND gender(무관/남성) AND beneficiary(당사자). 소스 = `programs.json` 중 `region_scope ∈ {서울자치구, 서울전체}`.
- `진단[].한계` : `conflict_rules`(낙인·의존, Q=프로파일 quadrant) + `structural_limits`(자치구귀속·중복배제).
- `진단[].need` : `limit_to_needs[한계]` 그대로.
- `진단[].근거_논문` : `evidence.json` 중 `supports_ids` 가 트리거(DEP/STG)·need 코드를 포함하는 것, 관련도순 정예 4편.
- `이식후보` : `transplant_rule` — `fulfills_needs ∋ need` AND 선택 Q 무충돌 AND 선택 자치구 미시행. 소스 = `programs.json` 전체. `reference_priority`(국가→해외, 국내지역→국내타지역, 서울→서울타자치구)는 런타임 계산, 티어별 골고루.

## 3. 출력 예시 (LLM이 쓰는 결과)

```
[노원구 · 서울 5060 남성 1인가구 처방]

① 현행: 노원구는 현재 'P-0931 안부확인' 등으로 1인가구 안부를 살피고 있습니다.

② 한계: 그러나 이 제도들은 본인 신상이 식별되는 대면·신청 방식(STG_FaceExposure)에
   기대고 있어, 도움요청 자체를 '남자답지 못함'으로 내면화하는 5060 남성에게
   진입 장벽이 됩니다. WHO 유럽(2020)은 전통적 남성성 규범이 남성의 도움요청을
   구조적으로 억제한다고 보고했습니다.

③ 방향: 따라서 낙인을 자극하지 않는 전달방식(무낙인, NEED_NoStigma) —
   '복지 수혜자'가 아니라 '활동 주체'로 진입하는 설계가 필요합니다.

④ 외부 레퍼런스: 이 방향은 이미 검증됐습니다. 호주의 '남성들의 작업장(Men's Shed)'은
   남성이 복지 신청이 아니라 작업·취미 활동의 주체로 모이게 해 고립을 줄인 모델입니다.
   (해외 검증 사례)

⑤ 이식 제안: 따라서 노원구에도 '복지'가 아닌 '활동' 언어로 5060 남성이 모일 수 있는,
   신청·식별 부담 없는 관계형 제도가 마련돼야 합니다.
```

## 4. 구현 순서 (제일 쉬운 길)

1. **그래프 계산은 코드로** — `ontology.json` 규칙대로 (자치구 입력) → 위 입력 JSON 생성하는 파이썬 한 개.
   - evidence 매칭: `supports_ids ∩ {그 진단의 need/DEP/STG}` 가 비지 않으면 첨부.
2. **그 JSON을 위 SYSTEM 프롬프트와 함께 LLM에 던지기.**
3. LLM은 ①~⑤만 쓴다. (계산·검색 안 함 → 환각 차단)

> 지금은 그래프를 'DB/노드'로 안 만들어도 됨. 코드가 ontology 규칙으로 즉석 계산해서
> 입력 JSON만 만들어주면 충분. 추적·재사용 필요해지면 그때 노드化.
